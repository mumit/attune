# Deployment guide

*This is the self-hosted, long-running-service guide — covering both the
default polling track and its Google Pub/Sub push variant; see
[`modes.md`](modes.md) for how these fit alongside the hosted multi-tenant
service.*

Attune is a long-running Python service. One instance acts for one principal
and has its own Workspace credential or MCP endpoint, model credentials,
channel credentials, memory, state, and audit log. Run separate isolated
instances for separate principals.

This guide describes the current single-principal deployment. The normative
[security architecture](security-architecture.md) defines the controls for the
planned setup assistant and hosted service. In particular, local files, SQLite,
JSON, JSONL, and a Qdrant collection are not a multi-tenant security boundary.

There are two practical deployment tracks:

- **Polling on a standalone host or VM** is the recommended starting point.
  Gmail, Calendar, and optional Google Chat reads are outbound calls. Slack
  Socket Mode is also outbound-only. No public endpoint, Pub/Sub, or Cloud Run
  service is needed.
- **Google Cloud push ingestion** runs Attune on a Compute Engine VM and uses
  Pub/Sub for lower-latency Gmail/Chat events. A stateless Cloud Run
  republisher receives Calendar webhooks and optional Google Chat app
  interactions, then publishes count-free signals/events to Pub/Sub.

MCP deployments always use the first track: Attune polls the MCP server over
Streamable HTTP. Provider watches, if any, belong to that server.

Start with [`getting-started.md`](getting-started.md) and move to push only
after `attune doctor`, `attune brief`, Slack posting, and polling all work.
Use [`configuration.md`](configuration.md) for the complete environment-variable
reference; the deployment examples below intentionally show only the settings
relevant to each topology.

## 1. Deployment shape and trust boundaries

A complete direct-OAuth deployment can contain three distinct identities:

1. **Google authorized user** — grants Gmail and Calendar access for the
   principal. This is an authorized-user JSON generated by `attune init`, not
   the VM service account.
2. **Attune runtime service account** — lets a GCP VM pull Pub/Sub and read its
   Secret Manager entries. It does not impersonate the mailbox.
3. **Republisher service account** — lets Cloud Run publish only to the
   Calendar and Chat-interaction topics. It holds no mailbox OAuth, model key,
   memory, or workflow state.

Google Chat adds a fourth identity: a dedicated Chat app service account with
the `chat.bot` scope for proactive app-authored messages. Never reuse it as the
Gmail/Calendar credential.

With MCP, the MCP service replaces the first identity and owns provider
credentials and consent. The Attune process only needs its MCP endpoint/token.

## 2. Polling on a standalone server or VM

### Install and configure

Install Docker, Git, and Python 3.11 or newer (3.12 recommended). Use a
dedicated OS account and directory; the exact package-manager commands vary by
distribution:

```bash
sudo useradd --system --create-home --shell /usr/sbin/nologin attune
sudo mkdir -p /opt/attune /var/lib/attune /etc/attune
sudo chown -R attune:attune /opt/attune /var/lib/attune

sudo -u attune git clone <repository-url> /opt/attune/source
sudo -u attune python3 -m venv /opt/attune/venv
sudo -u attune /opt/attune/venv/bin/pip install --upgrade pip
sudo -u attune /opt/attune/venv/bin/pip install \
  -e "/opt/attune/source[orchestrator,memory,google,slack,mcp]"
```

Start Qdrant from the checkout:

```bash
cd /opt/attune/source
docker compose -f deploy/compose.yml up -d
```

On an interactive host, `attune init --target local` combines environment
editing, a previewed Qdrant Compose apply, resumable setup state, and the full
Doctor validation. The manual command remains useful for a dedicated service
account or configuration-management system. The local target deliberately
does not install or retain cloud-administrator credentials.
Use `attune status --check` for combined setup and live health, or
`attune repair` to preview, reapply, and revalidate the recorded local plan.
Repair refuses an unrecorded deployment rather than guessing resource ownership.

Alternatively, run the whole stack in containers after a `.env` exists:

```bash
docker compose -f deploy/compose.yml --profile assistant up -d --build
```

The assistant container sees its named state volume as `/data`; it cannot see
an arbitrary credential path from the host. Put the authorized-user JSON in
that volume or add a read-only bind mount, and set
`ATTUNE_GOOGLE_CREDENTIALS_FILE` to the **container** path.

For a direct-OAuth host, generate the authorized-user JSON interactively on a
machine with a browser using `attune init`, then transfer it over an encrypted
channel to `/var/lib/attune/google_authorized_user.json`. Set mode 0600 and
ownership to the service account. Do not deploy the downloaded desktop-client
JSON as though it were the resulting authorized-user credential.

For MCP, install the `mcp` extra, use polling, and configure one shared
Streamable HTTP URL or separate Gmail/Calendar URLs. The server must implement
[`mcp-contract.md`](mcp-contract.md).

Create `/etc/attune/attune.env`, owned by `attune` and mode 0600. A minimal
direct-OAuth polling configuration is:

```dotenv
ATTUNE_WORKSPACE_BACKEND=google_oauth
ATTUNE_INGESTION_MODE=poll
ATTUNE_USER_ID=owner@example.com
ATTUNE_INTERNAL_DOMAINS=example.com
ATTUNE_DATA_DIR=/var/lib/attune
ATTUNE_GOOGLE_CREDENTIALS_FILE=/var/lib/attune/google_authorized_user.json
GOOGLE_PROJECT_ID=my-google-project

ATTUNE_LLM_BASE_URL=https://api.openai.com/v1
ATTUNE_LLM_API_KEY=...
ATTUNE_MODEL_DEFAULT=...
ATTUNE_EMBEDDING_BASE_URL=https://api.openai.com/v1
ATTUNE_EMBEDDING_API_KEY=...
ATTUNE_EMBEDDING_MODEL=text-embedding-3-small
ATTUNE_EMBEDDING_DIMENSIONS=1536

ATTUNE_QDRANT_HOST=127.0.0.1
ATTUNE_QDRANT_PORT=6333
ATTUNE_TIMEZONE=America/Vancouver
ATTUNE_BRIEF_TIME=07:30
```

For MCP, replace the Workspace block with:

```dotenv
ATTUNE_WORKSPACE_BACKEND=mcp
ATTUNE_INGESTION_MODE=poll
ATTUNE_MCP_URL=https://workspace-mcp.example.com/mcp
ATTUNE_MCP_TOKEN=...
```

Append the Slack configuration from
[`getting-started.md`](getting-started.md#6-create-the-slack-app) if Slack is a
selected route.

### Run under systemd

Create `/etc/systemd/system/attune.service`:

```ini
[Unit]
Description=Attune workspace assistant
Wants=network-online.target
After=network-online.target docker.service

[Service]
Type=simple
User=attune
Group=attune
WorkingDirectory=/opt/attune/source
EnvironmentFile=/etc/attune/attune.env
ExecStart=/opt/attune/venv/bin/attune run
Restart=on-failure
RestartSec=10
NoNewPrivileges=true
PrivateTmp=true

[Install]
WantedBy=multi-user.target
```

`attune run` performs its fatal Doctor checks before starting. To run the full
Doctor battery with the same systemd environment, then start the service:

```bash
sudo systemd-run --wait --pipe --uid=attune \
  --property=EnvironmentFile=/etc/attune/attune.env \
  --working-directory=/opt/attune/source \
  /opt/attune/venv/bin/attune doctor
sudo systemctl daemon-reload
sudo systemctl enable --now attune
journalctl -u attune -f
```

Keep Qdrant's port private to the host or trusted network; do not add a public
firewall rule for port 6333.

## 3. Google Cloud push deployment overview

The GCP track uses:

- one Compute Engine VM for `attune run` and Qdrant;
- one Cloud Run republisher for Calendar and optional Google Chat callbacks;
- Pub/Sub topics/subscriptions;
- Secret Manager for model, Slack, and Google authorized-user credentials;
- one runtime service account and one republisher service account.

The credential-holding VM opens no inbound application port. It initiates
Pub/Sub pulls and Slack Socket Mode connections.

Set shell variables before following the commands:

```bash
export PROJECT_ID=my-attune-project
export REGION=us-central1
export ZONE=us-central1-a
gcloud config set project "$PROJECT_ID"
```

Billing must be enabled, and the operator needs permission to enable services,
create IAM/service accounts, Pub/Sub resources, Compute Engine instances,
secrets, builds, and Cloud Run services.

## 4. Enable Google Cloud and Workspace APIs

For Gmail and Calendar push plus the runtime infrastructure:

```bash
gcloud services enable \
  gmail.googleapis.com \
  calendar-json.googleapis.com \
  pubsub.googleapis.com \
  secretmanager.googleapis.com \
  compute.googleapis.com \
  run.googleapis.com \
  cloudbuild.googleapis.com \
  artifactregistry.googleapis.com \
  iam.googleapis.com
```

If Google Chat is enabled, also enable:

```bash
gcloud services enable \
  chat.googleapis.com \
  workspaceevents.googleapis.com
```

The OAuth project, consent screen, scopes, and authorized-user credential are
created exactly as described in
[`getting-started.md`](getting-started.md#4a-direct-google-oauth). Push changes
the transport, not the mailbox identity or scopes.

## 5. Create least-privilege service accounts

Create the VM identity and let it pull Pub/Sub:

```bash
gcloud iam service-accounts create attune-runtime \
  --display-name="Attune runtime"

gcloud projects add-iam-policy-binding "$PROJECT_ID" \
  --member="serviceAccount:attune-runtime@${PROJECT_ID}.iam.gserviceaccount.com" \
  --role="roles/pubsub.subscriber"
```

Create the Cloud Run identity:

```bash
gcloud iam service-accounts create attune-republisher \
  --display-name="Attune webhook republisher"
```

Grant Secret Manager access per secret rather than at project level where
possible. Google documents `roles/secretmanager.secretAccessor` as the role
that reads secret payloads and recommends granting it at the lowest practical
resource level: [Secret Manager access control](https://cloud.google.com/secret-manager/docs/access-control).

## 6. Store and grant secrets

Generate `google_authorized_user.json` before uploading it. Create secrets from
files/stdin so values do not appear in command history:

```bash
gcloud secrets create attune-google-authorized-user \
  --data-file="$HOME/.attune/google_authorized_user.json"

printf '%s' "$ATTUNE_LLM_API_KEY_VALUE" | \
  gcloud secrets create attune-llm-api-key --data-file=-
printf '%s' "$ATTUNE_EMBEDDING_API_KEY_VALUE" | \
  gcloud secrets create attune-embedding-api-key --data-file=-
printf '%s' "$SLACK_BOT_TOKEN_VALUE" | \
  gcloud secrets create attune-slack-bot-token --data-file=-
printf '%s' "$SLACK_APP_TOKEN_VALUE" | \
  gcloud secrets create attune-slack-app-token --data-file=-
```

Omit secrets for unused channels or for a shared model/embedding key. Grant the
VM identity access to each secret it uses:

```bash
for secret in \
  attune-google-authorized-user \
  attune-llm-api-key \
  attune-embedding-api-key \
  attune-slack-bot-token \
  attune-slack-app-token; do
  gcloud secrets add-iam-policy-binding "$secret" \
    --member="serviceAccount:attune-runtime@${PROJECT_ID}.iam.gserviceaccount.com" \
    --role="roles/secretmanager.secretAccessor"
done
```

If a listed secret was intentionally omitted, omit it from the loop too.

## 7. Create Pub/Sub topics and pull subscriptions

Gmail and Calendar need these pairs:

```bash
for name in gmail calendar; do
  gcloud pubsub topics create "attune-${name}"
  gcloud pubsub subscriptions create "attune-${name}-sub" \
    --topic="attune-${name}" \
    --ack-deadline=60
done
```

Gmail publishes directly to its topic. Google requires its system service
account to have Publisher:

```bash
gcloud pubsub topics add-iam-policy-binding attune-gmail \
  --member="serviceAccount:gmail-api-push@system.gserviceaccount.com" \
  --role="roles/pubsub.publisher"
```

This identity and requirement are documented in Google's
[Gmail push guide](https://developers.google.com/workspace/gmail/api/guides/push).

If Google Chat is in use, create the verified HTTP interaction path used for
app messages and card clicks:

```bash
gcloud pubsub topics create attune-chat-interaction
gcloud pubsub subscriptions create attune-chat-interaction-sub \
  --topic=attune-chat-interaction \
  --ack-deadline=60
```

Grant the republisher its Calendar topic, plus the Chat-interaction topic only
when Chat is configured:

```bash
gcloud pubsub topics add-iam-policy-binding attune-calendar \
  --member="serviceAccount:attune-republisher@${PROJECT_ID}.iam.gserviceaccount.com" \
  --role="roles/pubsub.publisher"

# Optional: only when Google Chat is configured.
gcloud pubsub topics add-iam-policy-binding attune-chat-interaction \
  --member="serviceAccount:attune-republisher@${PROJECT_ID}.iam.gserviceaccount.com" \
  --role="roles/pubsub.publisher"
```

As an alternative message-ingestion path, Attune can create a Workspace Events
subscription using `ATTUNE_CHAT_PUBSUB_TOPIC`. That separate topic must grant
Publisher to `chat-api-push@system.gserviceaccount.com`, as described in
Google's [Chat Pub/Sub setup](https://developers.google.com/workspace/chat/quickstart/pub-sub).
Do not enable both the HTTP app-message path and Workspace Events for the same
space unless duplicate delivery has been deliberately accounted for. Card
clicks still require the verified HTTP interaction path.

The runtime service account's project-level Subscriber role allows it to pull
the subscriptions. Narrow that role to individual subscriptions if your IAM
policy requires tighter isolation.

## 8. Deploy the Cloud Run republisher

`deploy/republisher` is a standalone Flask service with its own Dockerfile and
dependencies. It has two routes:

- `/calendar-webhook` receives Google Calendar's header-only HTTPS signal and
  republishes it to `attune-calendar`. The runtime treats it only as a prompt to
  reconcile against the authenticated Calendar API; webhook content is never
  trusted as event data.
- `/chat-interaction` verifies Google Chat's bearer token, handles dialog-open
  responses synchronously, and republishes messages or decisions to
  `attune-chat-interaction`. It is unnecessary for a Slack-only deployment
  except that the Calendar route still uses the same service.

Run its independent test suite before deployment:

```bash
python -m pip install -r deploy/republisher/requirements.txt pytest
pytest deploy/republisher/test_main.py -q
```

Deploy once to obtain the stable service URL:

```bash
gcloud run deploy attune-republisher \
  --source=deploy/republisher \
  --region="$REGION" \
  --service-account="attune-republisher@${PROJECT_ID}.iam.gserviceaccount.com" \
  --set-env-vars="CALENDAR_PUBSUB_TOPIC=projects/${PROJECT_ID}/topics/attune-calendar" \
  --allow-unauthenticated

export REPUBLISHER_URL="$(gcloud run services describe attune-republisher \
  --region="$REGION" --format='value(status.url)')"
```

When Google Chat is configured, set its topic and the exact audience URL used
by the verifier:

```bash
gcloud run services update attune-republisher \
  --region="$REGION" \
  --update-env-vars="CHAT_INTERACTION_PUBSUB_TOPIC=projects/${PROJECT_ID}/topics/attune-chat-interaction,CHAT_APP_AUDIENCE=${REPUBLISHER_URL}/chat-interaction"
```

For a Slack-only deployment, omit the Chat topic, audience, and update command.

Cloud Run source deployment uses `gcloud run deploy --source`; see Google's
[source deployment guide](https://cloud.google.com/run/docs/deploying-source-code).
The service must accept unauthenticated network requests because Calendar is
not a Cloud Run IAM principal. Chat requests are authenticated inside the
service. An organization policy that forbids public Cloud Run invocation needs
an approved alternative public webhook front end.

Google Calendar requires a public HTTPS callback and sends change notifications
as header-only POSTs; see the
[Calendar push guide](https://developers.google.com/workspace/calendar/api/guides/push).

## 9. Optional Google Chat app setup

Slack can remain the only channel. If Google Chat is also selected:

1. Create a service account for the Chat app and download its JSON key. This is
   configured as `ATTUNE_CHAT_CREDENTIALS_FILE` and is separate from the
   principal's authorized-user JSON.
2. Configure the Google Chat API app, enable interactive features, and add the
   app to the intended space.
3. Set its HTTP endpoint URL to
   `${REPUBLISHER_URL}/chat-interaction`.
4. Choose **HTTP endpoint URL** as the Authentication Audience and use the
   exact same URL for `CHAT_APP_AUDIENCE`.
5. Record the `spaces/...` resource and allowed `users/...` IDs.

Google's [request verification guide](https://developers.google.com/workspace/chat/verify-requests-from-chat)
explains the bearer token and audience modes. The republisher implements the
HTTP-endpoint audience path.

The principal's authorized-user credential needs the optional Chat read/event
scopes only when Chat polling or Workspace Events are used:

```text
https://www.googleapis.com/auth/chat.messages
https://www.googleapis.com/auth/chat.spaces.readonly
```

`attune init` currently requests only the core Gmail/Calendar scopes. If the
optional user-authenticated Chat ingestion path is selected, regenerate the
authorized-user file with the core scopes plus these Chat scopes before
enabling it. The Chat app service-account credential remains separate.

## 10. Create the Compute Engine VM

A small VM is sufficient for personal-scale traffic; use `e2-medium` as a
conservative starting point:

```bash
gcloud compute instances create attune-vm \
  --zone="$ZONE" \
  --machine-type=e2-medium \
  --service-account="attune-runtime@${PROJECT_ID}.iam.gserviceaccount.com" \
  --scopes=cloud-platform \
  --image-family=debian-12 \
  --image-project=debian-cloud \
  --boot-disk-size=30GB
```

Google recommends the `cloud-platform` VM scope and controlling access through
the attached service account's IAM roles: [Compute Engine service accounts](https://cloud.google.com/compute/docs/access/service-accounts).

On the VM, install Docker, Git, the Google Cloud CLI, and Python 3.11 or newer;
clone the repository; create `/opt/attune/venv`; and install:

```bash
/opt/attune/venv/bin/pip install \
  -e "/opt/attune/source[orchestrator,memory,google,slack]"
cd /opt/attune/source
docker compose -f deploy/compose.yml up -d
```

Use a separate persistent disk or a backed-up boot-disk directory for
`ATTUNE_DATA_DIR` and the Qdrant Docker volume. The source checkout should be
replaceable.

## 11. Materialize credentials and configure push mode

Attune does not fetch Secret Manager values itself. During provisioning (and
after rotation), materialize them into root/service-account-readable files and
restart the service. For example, on the VM:

```bash
sudo install -d -o attune -g attune -m 700 /var/lib/attune /etc/attune

gcloud secrets versions access latest \
  --secret=attune-google-authorized-user | \
  sudo tee /var/lib/attune/google_authorized_user.json >/dev/null
sudo chown attune:attune /var/lib/attune/google_authorized_user.json
sudo chmod 600 /var/lib/attune/google_authorized_user.json
```

Build `/etc/attune/attune.env` with mode 0600. Secret values can be appended
from `gcloud secrets versions access`; ensure the command output is redirected
and never logged. For example:

```bash
sudo install -o attune -g attune -m 600 /dev/null /etc/attune/attune.env
{
  printf 'ATTUNE_LLM_API_KEY='
  gcloud secrets versions access latest --secret=attune-llm-api-key
  printf '\nATTUNE_EMBEDDING_API_KEY='
  gcloud secrets versions access latest --secret=attune-embedding-api-key
  printf '\nSLACK_BOT_TOKEN='
  gcloud secrets versions access latest --secret=attune-slack-bot-token
  printf '\nSLACK_APP_TOKEN='
  gcloud secrets versions access latest --secret=attune-slack-app-token
  printf '\n'
} | sudo tee -a /etc/attune/attune.env >/dev/null
sudo chown attune:attune /etc/attune/attune.env
sudo chmod 600 /etc/attune/attune.env
```

Omit unused entries and use the LLM key directly for embeddings when the
gateway shares credentials. The non-secret push configuration to append is:

```dotenv
ATTUNE_WORKSPACE_BACKEND=google_oauth
ATTUNE_INGESTION_MODE=google_pubsub
ATTUNE_USER_ID=owner@example.com
ATTUNE_INTERNAL_DOMAINS=example.com
ATTUNE_DATA_DIR=/var/lib/attune
ATTUNE_GOOGLE_CREDENTIALS_FILE=/var/lib/attune/google_authorized_user.json
GOOGLE_PROJECT_ID=my-attune-project

ATTUNE_GMAIL_PUBSUB_TOPIC=projects/my-attune-project/topics/attune-gmail
ATTUNE_GMAIL_PUBSUB_SUBSCRIPTION=projects/my-attune-project/subscriptions/attune-gmail-sub
ATTUNE_CALENDAR_PUBSUB_TOPIC=projects/my-attune-project/topics/attune-calendar
ATTUNE_CALENDAR_PUBSUB_SUBSCRIPTION=projects/my-attune-project/subscriptions/attune-calendar-sub
ATTUNE_CALENDAR_WEBHOOK_ADDRESS=https://SERVICE.run.app/calendar-webhook
ATTUNE_CALENDAR_ID=primary

ATTUNE_QDRANT_HOST=127.0.0.1
ATTUNE_QDRANT_PORT=6333
ATTUNE_TIMEZONE=America/Vancouver
ATTUNE_BRIEF_TIME=07:30
```

Add model/embedding and Slack settings from the local guide. If Chat is used,
also set:

```dotenv
ATTUNE_CHAT_CREDENTIALS_FILE=/var/lib/attune/chat-app-service-account.json
ATTUNE_CHAT_SPACE=spaces/AAAA...
ATTUNE_CHAT_ALLOWED_USERS=users/123...
ATTUNE_CHAT_INTERACTION_PUBSUB_TOPIC=projects/my-attune-project/topics/attune-chat-interaction
ATTUNE_CHAT_INTERACTION_PUBSUB_SUBSCRIPTION=projects/my-attune-project/subscriptions/attune-chat-interaction-sub
```

Set `ATTUNE_CHAT_PUBSUB_TOPIC` and its subscription only when intentionally
using the alternative Workspace Events path described in section 7.

Use full resource names. Replace `my-attune-project` and `SERVICE.run.app` with
the actual values.

Install the systemd unit from section 2 and start the service. In push mode,
`Runtime.run()` registers configured watches/subscriptions at startup and the
built-in scheduler renews them. Gmail watches expire in at most seven days and
Google recommends daily renewal; Attune performs proactive scheduled renewal.

## 12. Slack on a server or GCP VM

The Slack app setup is identical on every host. Socket Mode does not require a
public endpoint, load balancer, firewall rule, signing secret, or Cloud Run
route. The VM only needs outbound HTTPS/WebSocket connectivity to Slack.

Use an owner-only `D...` destination where possible and configure:

```dotenv
SLACK_APP_TOKEN=xapp-...
SLACK_BOT_TOKEN=xoxb-...
ATTUNE_SLACK_CHANNEL=D0123456789
ATTUNE_SLACK_ALLOWED_USERS=U0123456789
ATTUNE_BRIEF_CHANNELS=slack
ATTUNE_APPROVAL_CHANNEL=slack
ATTUNE_NOTIFICATION_CHANNELS=slack
ATTUNE_INTERACTION_CHANNELS=slack
```

See the full creation/scopes/install sequence in
[`getting-started.md`](getting-started.md#6-create-the-slack-app).

## 13. Verify the deployment

Run these checks in order:

1. `attune doctor` reports PASS for installation, environment, data directory,
   model routes, workspace, and selected channels. Pub/Sub should pass in push
   mode and skip in polling mode.
2. `attune brief` reads Gmail and the current day's Calendar events.
3. `attune brief --post` reaches every configured brief channel.
4. Start `attune run` and inspect `journalctl -u attune -f`.
5. Send a new test email. Polling should report it after the poll interval;
   push should receive it through the Gmail subscription. Only actionable mail
   creates an approval card; noise is audit-only.
6. DM the Slack bot and confirm a reply.
7. Create overlapping Calendar events. Confirm one conflict notification and,
   where the connector supports it, a hold proposal. A normal appointment is
   brief-only.
8. Approve a Gmail draft in Slack and confirm the Gmail draft, pending state,
   memory signal, and audit event.
9. For push Calendar, inspect Cloud Run logs for the webhook and VM logs for
   reconciliation. Do not consume the production pull subscription with an
   `--auto-ack` diagnostic while Attune is running.

The first poll or watch registration establishes a baseline and intentionally
does not replay all pre-existing mailbox/calendar history.

## 14. Operations and maintenance

- **Logs:** monitor source failure warnings, five-minute loop heartbeats,
  `poll activity` lines, watch-renewal events, and retry state. Use
  `ATTUNE_LOG_JSON=1` with a structured logging collector.
- **Backups:** stop Attune or use a consistent filesystem snapshot. Back up
  `ATTUNE_DATA_DIR` and Qdrant together. State includes checkpoints, pending
  approvals, polling cursors, retry records, grants, conversation windows, and
  the append-only audit log.
- **Watch health:** a stopped push runtime cannot renew its own Gmail,
  Calendar, or Workspace Events watches. Alert on prolonged process downtime
  and renewal failures.
- **Secret rotation:** add a new Secret Manager version, rematerialize the
  local file/environment, restart Attune, run Doctor, perform a read-only smoke
  test, then disable the old version.
- **OAuth posture:** External/Testing credentials with restricted scopes are
  not durable production credentials. Monitor expiry and organizational app
  controls.
- **Upgrades:** deploy into a new virtual environment or image, run the full
  test suite and Doctor, stop the old process, preserve state volumes, and then
  start the replacement.
- **Isolation:** never share authorized-user JSON, Slack tokens, Qdrant
  collections, data directories, or Pub/Sub subscriptions between principals.

In polling mode, idle checks remain quiet apart from the heartbeat. Successful
activity is logged using counts only, for example
`poll activity: gmail=changed, 0 actionable`. This deliberately avoids putting
mail subjects, senders, event names, or message bodies in operational logs.
